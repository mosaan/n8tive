# アーキテクチャ説明書

#### 第1章 概要とアーキテクチャの定義 (Introduction and Architecture Definition)

この章では、ドキュメントの目的、システムの全体像、およびアーキテクチャに影響を与える要因を記述します。

*   **1.1. システムの責任範囲と境界:**
    *   提案するシステムが何を含み、何を含まないかを明確にし、システムと相互作用する外部のアクターやシステムとのインターフェイスの一般的な性質を記述します。

*   **1.2. アーキテクチャに影響を与える要因（制約と原動力）:**
    *   アーキテクチャの選択に影響を及ぼす要因を記述します。これには、**機能外要求**（性能、信頼性、可用性など）、**ソフトウェアを実行するプラットフォーム**（OS、DB、ネットワーク通信プロトコル）、**再利用可能な構築ブロック**（フレームワーク、市販ソフトウェアなど）、および採用する**標準や会社の方針**などが含まれます。
    *   （方向づけフェーズの焦点）特に、プラットフォームやミドルウェアの選択など、アーキテクチャに影響を及ぼす**重要な機能外要求**を記述します。

*   **1.3. 採用するアーキテクチャのスタイル/パターン:**
    *   システムを構築する際に採用する全体構造のスタイル（例：階層化アーキテクチャ、クライアント/サーバーなど）を記述します。

#### 第2章 機能とユースケースのビュー (Functional and Use-Case View)

この章では、アーキテクチャの妥当性を評価するための主要な機能要求を記述します。

*   **2.1. ユースケースモデルのビュー:**
    *   システムが提供する**重要で不可欠な機能**を記述するユースケース、または**ソフトウェアライフサイクルの初期に開発しなければならない重要な要求**を伴うユースケースを含めます。
    *   （方向づけフェーズの焦点）**システムの中心的な機能を形成する鍵となるユースケース**（全体のわずか5%～10%程度）、および**致命的リスクの理解に関わるユースケース**の一部を記述します。
    *   このビューには、各ユースケースの**詳細記述**の全体が含まれる場合があります。

#### 第3章 概念的構造のビュー（分析モデルのビュー）

この章では、アーキテクチャの論理的な構造を記述します。

*   **3.1. 分析モデルの概要と分割:**
    *   分析モデルのアーキテクチャビューは、分析モデルを分析パッケージとその依存関係に分割したものを含みます。
    *   （方向づけフェーズの焦点）**最初の荒削りの分析モデル**を記述します。これはアーキテクチャ候補の基礎となるものを理解するための最初の一歩となります。

*   **3.2. 中心的な分析クラス:**
    *   問題領域の重要な現象をカプセル化する**エンティティクラス**、重要な通信/ユーザーインターフェイスのメカニズムをカプセル化する**バウンダリクラス**、重要なシーケンスをカプセル化する**コントロールクラス**など、中心的な分析クラスを記述します。

#### 第4章 実現構造のビュー（設計モデルのビュー）

この章では、実装の青写真として機能するシステムの具体的な構造を記述します。

*   **4.1. 設計サブシステムとインターフェイス:**
    *   システムの基本的な構造を構成する**サブシステム、サービスサブシステム**、およびそれらの間に定義された**インターフェイス**を記述します。

*   **4.2. アーキテクチャ上重要な設計クラスとアクティブクラス:**
    *   アクティブクラス、汎用の設計メカニズムを表す設計クラス、およびアーキテクチャ上重要な分析クラスから由来を追跡できる設計クラスを記述します。

*   **4.3. 汎用の設計メカニズム:**
    *   永続性、透過的なオブジェクト分散、トランザクション管理、セキュリティ特性など、共通する要求を取り扱うために採用された**汎用の設計メカニズム**と、その実現方法（サブシステムやクラスによる）を記述します。

*   **4.4. 重要なユースケースの実現（コラボレーション）:**
    *   アーキテクチャ上重要なユースケースが、相互作用するサブシステムやクラスによってどのように実現されるかを記述します。

#### 第5章 物理配置のビュー（配置モデルのビュー）

この章では、システムが実行される物理的な環境を記述します。

*   **5.1. ノードとネットワーク構成:**
    *   システムを分散する**ノード**（プロセッサなどのハードウェアユニット）とその**接続**、および通信プロトコルなどの特徴を記述します。

*   **5.2. コンポーネントとアクティブクラスの配置:**
    *   アクティブクラスや実行可能コンポーネントが、どのノードにマッピングされ、そこで実行されるかを記述します。これはシステムの並行性と分散性を決定します。

#### 第6章 実装のビュー (Implementation View)

この章では、実装要素とアーキテクチャとの関係を記述します。

*   **6.1. アーキテクチャ上重要なコンポーネント:**
    *   実行可能コンポーネントや、汎用の設計メカニズムを実装したコンポーネントなど、アーキテクチャを理解する上で重要な**実装コンポーネント**を記述します。
    *   これらのコンポーネントが、対応する設計クラスや設計サブシステムにどのように追跡されるかを記述します。

*   **6.2. 実装モデルの構造:**
    *   （通常、重要性が低いが、必要に応じて）実装サブシステム、そのインターフェイス、およびコンポーネント間の依存関係を記述します。

### 各フェーズにおけるアーキテクチャ説明書の役割

| フェーズ | 文書の状態と焦点 |
| :--- | :--- |
| **方向づけフェーズ** | **初期の草案（アーキテクチャ候補の説明書）**。第1章、第2章、第4章（サブシステム/クラスのアウトライン、汎用メカニズムの概略）、第5章（ノードの概略）の**アーキテクチャ上重要な要素**に焦点を絞る。 |
| **推敲フェーズ** | **安定したアーキテクチャ説明書が完成**し、開発チーム全体の「標準」となる。第3章（分析モデルの構造）、第4章（サブシステムとインターフェイス）、第5章（詳細な配置）が充実し、**実行可能なアーキテクチュラルベースライン**を完全に説明する。 |
| **作成フェーズ** | アーキテクチャ説明書は**維持と最小限の更新**が中心。アーキテクチャの整合性を維持するために、必要に応じて変更点を反映させる。文書量自体は大きく増加しない。 |
| **移行フェーズ** | 最終製品のベースラインとして維持され、必要に応じて小さな欠陥修正に伴う変更を反映する。**プロジェクト完了後の保守の指針**となる。 |
